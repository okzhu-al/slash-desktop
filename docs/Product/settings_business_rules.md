# Slash 设置系统与后台 AI 服务产品业务规则文档

> [!NOTE]
> 本文档基于第一性原理，由资深产品经理对 Slash Desktop（基于 Tauri + React + Rust 架构）设置系统与后台 AI 服务核心模块进行代码反向拆解所还原的业务规则与逻辑设计。

---

## 1. 基础设置系统业务规则

Slash 在基础设置上恪守极致体验与高内聚设计，包含外观主题、显示语言及编辑器宽度的配置规则与交互逻辑。

### 1.1 外观主题模式自适应规则
*   **三种运行模式**：浅色模式 (`light`)、深色模式 (`dark`) 和系统跟随模式 (`system`)。
*   **持久化存储**：用户选择的主题类型将被持久化至前端 `localStorage`，键名为 `slash-theme`。
*   **切换与渲染机制**：
    *   当选择 `light` 或 `dark` 时，系统将直接向文档根节点 `document.documentElement`（即 `<html>` 标签）添加对应的 `light` 或 `dark` 样式类名，同时移除另一个，从而触发 Tailwind CSS 的 `dark:` 全局样式树切换。
    *   当选择 `system` 时，系统在初始化及切换时通过 `window.matchMedia("(prefers-color-scheme: dark)")` 动态检测 macOS 系统的日夜主题偏好。
*   **实时响应日夜变换**：
    *   在 `system` 模式下，系统会挂载一个全局实时媒体查询监听器（`MediaQueryListener`）。
    *   当 macOS 系统在日落或定时任务触发日夜主题切换时，Slash 能够在不重启、不刷新的情况下，自动调整根节点的 Class 映射，实现无缝平滑的主题跟随。

### 1.2 显示语言切换与热更新规则
*   **支持语言**：简体中文 (`zh-CN`) 与 英文 (`en`)。
*   **交互规则**：
    *   用户在语言栏切换时，系统调用 `i18n.changeLanguage(lang)` 触发整个应用界面文字的动态热替换，无需重新加载窗口。
    *   选定的语言代码会被同步写入 `localStorage` 的 `i18nextLng` 键中，供应用下次冷启动时作为首选语言直接加载。

### 1.3 编辑器显示宽度响应式排版规则
*   **显示规格**：标准宽度 (`standard`) 与 通栏宽度 (`full`)。
*   **设计原则**：
    *   `standard`：限制最大宽度为 `max-w-3xl`（居中排版），符合人类阅读习惯及中段长度写作的视线生理特征，保护专注度。
    *   `full`：取消宽度限制 (`max-w-none`)，允许编辑器横向铺满整个窗口，主要用于大屏设备展示多列复杂表格、思维导图式混合大纲或极客全景编辑场景。
*   **持久化机制**：保存在 `localStorage` 的 `slash-theme-width` 中，由全局 `ThemeProvider` 提供 Context 上下文状态，编辑器容器组件响应式读取渲染。

## 2. 多库 (Vault) 管理业务规则

Slash 支持多本地知识库（Vault）的并行管理、快速切换与冷启动自动加载。

### 2.1 存储与历史记录管理规则
*   **存储位置**：所有库的使用历史以 JSON 数组形式持久化保存在前端的本地存储 `localStorage` 中，键名为 `slash-vaults-history`。
*   **数据实体结构**：
    ```typescript
    interface VaultInfo {
        path: string;       // 库在本地文件系统中的绝对路径
        name: string;       // 库显示名称（自动截取路径的最后一级 basename，若为空则降级显示 "Untitled Vault"）
        lastUsed: number;   // 最后一次被激活打开的时间戳（毫秒级）
    }
    ```
*   **冷启动自动加载**：应用冷启动时，系统自动检索历史列表中 `lastUsed`（最后使用时间戳）最大的一条，获取其对应的绝对路径作为**主库**静默加载。

### 2.2 macOS 不区分大小写排重规则
*   **背景约束**：macOS 默认的文件系统（APFS）是**不区分路径字符大小写**的（Case-Insensitive）。
*   **去重逻辑**：
    *   在加载、写入或更新历史库列表时，系统会强制将库绝对路径转为小写字符（`path.toLowerCase()`）进行全局唯一性校验。
    *   若检测到相同的绝对路径（仅大小写差异），系统会**强行合并排重，且仅保留最后修改时间戳（`lastUsed` 最大）的那条记录**。
    *   去重后，系统会自动重写 `localStorage`，保证历史列表中不会出现逻辑重复的物理路径。

### 2.3 物理存在性校验与“幽灵库”静默清理
*   **场景描述**：当用户在设置面板或库切换列表进行操作时，系统为了避免用户点击已被操作系统物理删除、移动或外置硬盘拔出的无效库路径，设计了静默校验。
*   **检测机制**：
    *   系统会对历史列表中的每一个路径，调用 Tauri 顶层物理文件系统 API `exists(path)` 进行真实性检验。
    *   如果物理检测返回 `false`（路径不存在），系统将触发**静默垃圾清理（Silent Garbage Collection）**，自动从 `localStorage` 中物理剔除该条数据。
    *   如在校验时因权限或不可抗力发生异常，为防误删，系统将保守地选择保留该条目。

### 2.4 新建/导入库的物理结构初始化要求
*   无论用户是通过“新建库（`createVault`）”还是“导入库（`openVault`）”，在选择本地文件夹后，系统都会强制执行以下初始化逻辑：
    1.  调用本地系统 IO，在所选目录根路径下检测或强行创建一个名为 **`.slash`** 的隐藏配置存储文件夹。
    2.  若隐藏目录创建失败，系统将立刻阻止库的添加和后续连接，并抛出 `Could not initialize vault configuration.` 致命异常。
    3.  `.slash` 文件夹将作为本地 SQLite 数据库文件（`slash.db`）、用户自定义技能（`skills/`）和各类临时配置的物理基座，保证了数据对 Vault 的物理聚合和可便携移动性。

## 3. 自定义快捷键系统业务规则

Slash 包含一套完整的快捷键自定义捕获、防冲突、作用域沙箱隔离与重置规则。

### 3.1 快捷键捕获与跨平台标准化映射机制
*   **捕获阶段交互**：
    *   在快捷键自定义面板中，当用户点击某个快捷键时，该项进入编辑状态，显示 `animate-pulse` 动态呼吸闪烁状态，提示 `Press new key...`（按新键...）。
    *   **拦截单一修饰键**：系统注册 `onKeyDown` 捕获底层按键事件。当用户仅单按修饰键（如 `Control`, `Meta`（Mac ⌘ 键）, `Alt`, `Shift`）时，系统判定为无效输入并予以拦截，不予输出。
*   **跨平台标准化组合拼接（`captureKeyCombo`）**：
    *   系统通过捕获组合事件状态，利用 `Mod` 概念实现 Windows/Linux 与 macOS 的平权转换。
    *   `metaKey`（Mac ⌘ 键）或 `ctrlKey`（Ctrl 键）会被统一重映射为抽象组合词 **`Mod`**。
    *   其他修饰键映射为 `Alt`、`Shift`。
    *   普通按键字符一律强制转为大写（如字母 `k` 转为 `K`），空格键特殊字符映射为 `SPACE`。
    *   最终拼接成以 `+` 串联的标准化字符串：`Mod+Alt+K`。
    *   **退出/取消拦截**：在编辑状态下，如果用户按 `Escape` 键，或者焦点失去触发 `onBlur` 事件，系统将无害退出捕获模式，放弃保存。

### 3.2 基于 Scope 作用域沙箱的场景隔离规则
为了防止快捷键产生操作污染，快捷键系统内置了 Scope 作用域沙箱机制。
*   **作用域定义**：
    *   `global`：全局级快捷键作用域。例如 `New Note`（新建笔记 `Mod+N`）、`New Folder`（新建文件夹 `Mod+Shift+N`）、`Find in Note`（全局搜索 `Mod+F`）等。
    *   `editor`：编辑器级快捷键作用域。例如 `Toggle Bold`（粗体 `Mod+B`）、`Toggle Italic`（斜体 `Mod+I`）、`Insert Table`（插入表格 `Mod+Shift+T`）等。
*   **激活逻辑**：
    *   系统内置一个活跃作用域集合 `activeScopes = Set<Scope>`，默认包含 `global`。
    *   当编辑器未获得焦点时，`activeScopes` 仅包含 `global`，所有编辑器专属排版组合键即使在物理上被按下也会被**沙箱隔离阻断**。
    *   当编辑器聚焦且用户处于输入状态时，系统将 `editor` 作用域加入 `activeScopes` 集合，排版快捷键解锁。

### 3.3 按键冲突防爆匹配规则
*   **背景冲突**：若用户自定义快捷键时，强行把两个不同功能的命令绑定了完全相同的物理按键（如将 `app.new_note` 与 `editor.bold` 均设为了 `Mod+N`）。
*   **匹配优先顺序匹配阻断**：
    *   系统在 `handleKeyDown` 全局捕获事件触发后，会对标准化后的组合键进行 `COMMANDS` 注册表检索。
    *   **由于按键匹配采用的是 `Object.values(COMMANDS).find` 先到先得的单向匹配策略，所以在注册表中排序靠前的命令会强行吞掉按键，直接拦截执行，而排在后面的同名组合键将永远被冷冻阻断。**

### 3.4 持久化与一键/单项出厂默认降级规则
*   **持久化机制**：快捷键自定义由 Zustand 架构驱动，自动将修改后的 `customKeys: Record<string, string>` 字典同步到前端本地存储的 `slash-keybindings` 中。
*   **读取优先级重定向（`getEffectiveKey`）**：
    *   系统在读取某个功能的有效快捷键时，遵循**自定义覆盖默认**原则：
        $$\text{Effective Key} = \text{customKeys}[id] \ \text{??} \ \text{DefaultPresetKey}$$
        若 `customKeys` 存在映射，则使用用户修改值；若无，则优雅重定向降级至注册表 `COMMANDS` 里的默认出厂预设组合。
*   **单项重置**：用户在列表右侧点击“重置”按钮，系统触发 `resetKey` 物理删除 `customKeys` 字典中该命令的 Key-Value 映射。此时，由于读取优先级自动重定向，该快捷键瞬间安全回归出厂默认值。
*   **一键还原**：通过 `resetAllKeys` 物理清空整个 `customKeys` 字典，使整个软件的所有自定义快捷键全部回归出厂状态。

## 4. AI 设置与运行时连接管理业务规则

Slash 支持本地大模型与在线大模型多提供商的无缝混合驱动，并在设置中实现了极致稳健的配置与回退方案。

### 4.1 本地 Ollama 大模型拉取、就绪校验与异步进度推送
*   **就绪检测超时限制（`check_ollama_model`）**：
    *   当在设置中加载或切换本地 Ollama 模型时，为了防止 Ollama 后台服务僵死或未启动导致主应用卡死，Rust 后端在执行 `client.list_local_models()` 时强制注入了 **1500 毫秒的严格超时限制**：
        ```rust
        tokio::time::timeout(std::time::Duration::from_millis(1500), client.list_local_models())
        ```
    *   如果在 1.5 秒内未收到 Ollama 响应，系统直接判定为连接超时，向前端抛出友好网络故障警告，阻止死锁。
    *   **名称模糊匹配**：在校验本地是否拥有目标大语言模型时，系统同时比对模型全称及附加 `:latest` 后缀的版本（如匹配 `gemma2` 与 `gemma2:latest`），以提高检测成功率。
*   **拉取进度异步推送流 (`pull_ollama_model`)**：
    *   当用户点击下载某个本地模型（如 `qwen2.5:1.5b`）时，后端通过 `ollama_rs` 客户端的 `pull_model_stream` 接口建立长连接，以流（Stream）的形式异步接收下载字节进度。
    *   后端不断读取流数据包，将其包装为 `PullProgress` 统一数据实体（包含已下载字节 `completed`、总字节 `total`、当前百分比及报错状态）。
    *   通过 Tauri 事件机制以事件名 `"ollama:pull-progress"` 向 React 前端进行**毫秒级高频推送**，前端拦截事件并渲染高精度动画进度条，提供丝滑体验。

### 4.2 在线 AI 提供商多服务商 CRUD 规则与 Keychain 集成
*   **在线服务商定义**：系统提供包括 Google Gemini、DeepSeek、OpenAI、阿里通义千问（Qwen）等主流平台预设，以及自定义（Custom）兼容协议端点。
*   **存储去机密化原则（Keychain 安全存储）**：
    *   **在线提供商的 JSON 字典存放在 SQLite 的数据库表 `ai_settings` 中，但出于极高安全性要求，绝对不允许将 API Key 明文保存在数据库中！**
    *   所有的 API 密钥在提交时，都会通过 Rust 层的 Keychain 组件调用操作系统的安全钥匙串（Keychain）存储 API（`keychain::store_api_key_for(provider_id, api_key)`）进行硬件级安全加密隔离保护。
    *   如果系统在版本迁移时检测到旧版数据库 JSON 字段中残留明文 API 密钥，会自动将其抹去，并迁移到 Keychain 托管，实现双向保密升级。
*   **激活逻辑（`activate_online_provider`）**：
    *   当用户启用某个在线服务商时，系统从安全 Keychain 中加载其专属 API 密钥，将其与服务器端点（Base URL）和选定模型名注入主 `provider_config`。
    *   将 `provider_type` 设置为 `"online"`。
    *   通过 `AIService` 触发运行时热重构：即调用 `rebuild(config)`，在不断电不重启情况下即时重新构建后台 HTTP 请求客户端连接。

### 4.3 在线服务商删除与自动回退本地模式规则
如果用户在列表中删除某个在线服务商配置时，系统遵循以下回退保护链：
1.  **非激活状态删除**：若被删除的服务商当前未被用户激活，则系统仅物理抹除该服务商在数据库 JSON 列表中的配置项及 Keychain 密钥，主服务连接不受影响。
2.  **激活状态删除（回退到备用在线）**：若被删除的服务商是当前**正在生效**的 AI 驱动源：
    *   若列表里还残留其他可用的在线服务商，系统会自动**自动激活列表首项**，将其无缝应用，并热重构 `AIService` 连接。
    *   **激活状态删除（强行回退到本地 Ollama）**：若用户删除了最后一个在线提供商，导致列表为空，系统会自动触发**极致安全降级机制**：
        *   将主 AI 全局配置强制置为 `"local"` 模式。
        *   清空所有主配置中的在线 API 请求密钥与域名端点信息。
        *   触发 `service.rebuild`，强制在后台重新加载本地 Ollama 服务端点与模型。
        *   这保证了即使外网完全切断，本地核心的语义搜索、多维向量构建、快捷摘要及标签整理依然能完美安全运转，没有系统崩溃死锁风险。

### 4.4 文件夹级 AI 驱动局部覆盖机制（`.slash-folder.yml`）
为了提供细粒度的知识库大模型管控（如机密文件夹限制用本地大模型，公共速记文件夹用在线高速大模型），Slash 实现了独特的目录级覆盖：
*   **逐级向上爬寻检索（`resolve_config_for_path`）**：
    *   当系统处理某个笔记或文件夹的 AI 操作（如快速分类、生成大纲、向量化）时，Rust 后端会首先获取该笔记的物理相对路径。
    *   系统从当前笔记所在文件夹开始，**沿目录树逐级向上（Upward Traversal）** 检索是否存在 `.slash-folder.yml` 局部配置文件。
*   **属性局部覆盖**：
    *   一旦检测到配置文件，系统会立即解析该 YML 中定义的 `provider` 和 `model`。
    *   将此局部的 AI 服务配置动态注入当前运行线程的 `AIService` 内部，强行覆盖全局设置，重新构筑局部大模型网络客户端。
    *   这实现了极其高级的“按需匹配、按目录决策大模型驱动源”的精细化知识库运作。

## 5. Whisper 语音模型依赖的 Sidecar 服务的就绪检测与底层机制

Slash 内置了高精度语音识别功能（Whisper 语音模型），其后台转写引擎依托于一个独立封装的 Sidecar（基于 MarkItDown 协议扩展）子进程运行。

### 5.1 跨平台物理分发与静默安装机制
*   **跨平台二进制分发**：根据安装的宿主操作系统，系统自动定位与当前目标三元组相符的可执行包，Mac Arm 架构为 `slash-sidecar-aarch64-apple-darwin`，Mac Intel 为 `slash-sidecar-x86_64-apple-darwin`。
*   **确保安装逻辑（`ensure_sidecar_installed`）**：
    *   在应用启动的 Setup 黄金阶段，系统会自动检索用户系统的本地应用支撑目录：`~/Library/Application Support/Slash/sidecar/...`。
    *   若该目录不存在，或者目录内缺乏合法的版本标记文件 `version.json`，系统会判定 sidecar 发生物理缺失或受损。
    *   系统会自动锁定应用包 Bundle 中的内部集成资源 `Resources/binaries/`，调用 `copy_dir_recursive` 算法，将整个 sidecar 运行环境**静默、完整地复制安装**到本地应用支撑目录下，并在 Unix系统上赋予可执行权（`0o755` 权限）。

### 5.2 双向版本兼容性与静默升级/降级逻辑
*   为了防止应用升级后底层 sidecar 程序过旧导致接口崩溃，或应用降级后使用了不兼容的高版本 sidecar，系统设计了严苛的版本校验（`check_version_compatibility`）：
    *   系统同时读取**已安装 sidecar** 的 `version.json` 和**应用安装包内置（Bundle）**的 `version.json`。
    *   对比其中包含的两个字段：`sidecar_version`（Sidecar 物理版本）与 `min_app_version`（要求的最低 App 版本）。
*   **触发升级/降级判定**：
    *   若“当前 App 版本” **小于** “已安装 Sidecar 所要求的最低 App 版本”（发生应用版本降级），或者“应用 Bundle 内置的 Sidecar 版本” **大于** “当前已安装在本地支撑目录的版本”（发生应用升级）。
    *   系统会立即启动**静默覆盖修复**：物理删除掉本地 Application Support 目录下的旧文件夹，从 Bundle 重新全量复制，确保物理执行体版本与代码版本处于完美双向契合状态。

### 5.3 基于 stdout 协议的端口自动发现与就绪判定
*   **避免端口死锁**：为了防止多个 Slash 实例启动发生固定本地端口冲突，Whisper Sidecar 启动时不采用硬编码的静态端口，而是采用**动态端口扫描绑定机制**。
*   **就绪发现（`spawn_process`）**：
    *   Sidecar 启动并成功寻找并绑定可用随机端口后，会在其系统标准输出（`stdout`）中以行流打印一条明确的协议信号：`SIDECAR_READY:{port}`（例如 `SIDECAR_READY:53210`）。
    *   `SidecarManager` 拥有一个独立的子线程高频轮询读取子进程的 `stdout`，一旦成功捕获到带有 `SIDECAR_READY:` 前缀的行输出，即刻解析出端口号存入共享的 `port` 锁中，并将重启计数清零。
    *   此时，应用前后端即可安全通过 `http://127.0.0.1:{port}` 发起 HTTP 请求，判定 Whisper 语音转写服务就绪。

### 5.4 基于裸 TCP 连接手工构建 HTTP 请求的健康检查与指数退避重启
*   **轻量健康检查（`start_health_checker`）**：
    *   系统启动后，会常驻一个后台守护健康检查线程，每隔 **60 秒** 对已启动的 sidecar 执行一次 `/health` 接口探针检查。
    *   **极致轻量化架构**：为了避免在 Rust 的 Sidecar 守护进程管理中引入庞大的第三方异步 HTTP 库，系统利用标准库 `std::net::TcpStream` 建立了最底层的裸 TCP 连接。
    *   **手工拟态 HTTP 请求**：系统通过对 TCP 通道纯手工拼接、发送最原始的 HTTP GET 文本请求：
        ```http
        GET /health HTTP/1.0
        Host: 127.0.0.1:{port}
        
        ```
    *   然后只读取响应的前 64 字节，通过字符串比对判断是否包含字符 `200`（即 `HTTP/1.1 200 OK`），来轻量级断定 Sidecar 服务健康状况，极大地节省了系统算力与内存。
*   **僵死判定与崩溃重启**：
    *   当语音转写遇到极大音视频，或者模型加载阻塞导致健康检查**连续 5 次失败**（即 5 分钟无响应），系统直接判定 Sidecar 服务僵死。
    *   守护线程会自动调用系统的 `kill` 发送硬终止信号物理杀死该僵死进程，并触发自动重启。
*   **三层指数退避重启规则（`try_restart`）**：
    *   **最大重试次数限制**：为了防止因为配置受损、宿主系统环境缺失导致无限重启造成 CPU 熔毁，系统设定了最大自动重启次数上限 `MAX_RESTART_ATTEMPTS = 3`。
    *   **指数退避延迟（Exponential Backoff）**：
        为了给宿主机提供充分的物理资源释放时间，重启时间间隔采用指数级退避延迟：
        $$\text{Delay} = 1\text{s} \times 2^{\text{attempt} - 1}$$
        即：第一次重试在进程挂掉 1 秒后触发，第二次在 2 秒后触发，第三次在 4 秒后触发。
    *   若三次重启均告失败，系统将永久挂起，停止重试，直至用户手动重启软件重新初始化。

## 6. AI 技能模块（Magic Wand）核心业务规则

Slash Desktop 内置了六项高价值智能排版与语义发现工具，构成了 Slash 的核心知识生产力。

### 6.1 缓存去重与冷冻时间锁（Idempotence / Cooldown Cache）
为了在用户高频编辑或保存时，防止发起并行的、无谓的大模型 API 重复调用以节省 API 账单和系统 CPU 资源：
*   **数据模型缓存**：AI 模块所有的生成成果均持久化记录在本地 SQLite 的 `ai_metadata` 表中。
*   **物理哈希去重缓存锁（`ai_skill_state`）**：
    *   相较于传统内存缓存重启后即告丢失，Slash 会在数据库中建立 `ai_skill_state` 持久化缓存状态表。
    *   每次运行 AI 技能（如自动摘要、智能重命名、自动打标）前，计算当前笔记全文内容的字符哈希值 `content_hash = calculate_content_hash(&content)`。
    *   **判定拦截**：若数据库中记录的 `last_content_hash == content_hash` 且 `last_result == 'success'`，系统直接**熔断大模型网络请求**，以 `skipped` 状态快速返回，命令前端直接提取 `ai_metadata` 中上一次生成的持久化结果呈现。这大幅度降低了高频编辑产生的网络风暴。

### 6.2 智能任务队列与短文本熔断/清理策略
*   **双轨队列（`AIQueueManager`）**：
    *   所有 AI 任务在内核中被划分为 **`HeavyJob`（重度生成式 LLM 任务）** 与 **`FastJob`（快速文本/段落向量化嵌入任务）**，在并发队列中分类排队，通过 Oneshot Channel 调度，防止大模型响应缓慢引起界面线程阻塞卡死。
*   **短文本硬熔断策略（Circuit Breaker）**：
    *   **触发阈值**：提取笔记的有用正文（过滤掉多余的格式字符），计算有效字符数 `meaningful_char_count`。若其 **小于最低阈值 `MIN_CONTENT_FOR_AI = 50` 字符**：
        *   系统会立刻触发**短文本熔断**，直接拒绝对其运行任何大模型技能生成和向量嵌入流程。
    *   **历史垃圾物理清理**：
        *   为了防止文本被删减后，界面上仍然展示着这篇笔记以前很长时所产生的“过时 AI 数据”（如过时摘要或标签），在熔断触发时，系统会**强行执行垃圾清理事务**：
        *   从数据库 `ai_metadata`、`embeddings_v2`、`ai_suggestion_cache` 以及 `ai_skill_state` 中物理彻底删除这篇笔记所有的历史 AI 信息。
        *   向前端派发 `ai:cleared` 广播事件，让 UI 界面上的 AI 卡片同步清空，保证“所见即所得”。
*   **短文本排除例外（多模态媒体引用保留）**：
    *   若这篇很短的笔记中包含类似 `![image](assets/xxx)` 的媒体文件引用（如音视频速记、截图笔记），即使字数少于 50 字，系统也会**绕过熔断，不予清理**。而是移交给 `EmbeddingWorker` 进行富文本多模态向量化，确保多媒体的丰富语义资产不被遗漏。

### 6.3 Auto Summary (自动摘要) 业务规则
*   **技能输入策略**：
    *   采用 **`ProfileWithChunks`（画像加关键片段）** 的智能输入机制。
    *   将笔记的全文与通过编辑暗示（`EditHint`）得出的前 5 个最关键段落片段结合，构建出压缩版的 Skill 输入文本，防止超长文档撑爆 LLM 上下文（若超出上下文限制，会自动截取最高限制数）。
*   **生成与物理嵌入（P2 优先级向量）**：
    *   调用 LLM 产生一段 150 - 200 字的高精炼摘要，并将其更新到 FTS 全文索引及 `ai_metadata` 表的 `summary` 字段中。
    *   生成成功后，会立即在向量表中注册一条优先级为 **`P2` (Summary Embedding)** 的待向量化任务，随后由后台 worker 将此摘要转为向量，极大增强全局检索与关联分析的精确度。

### 6.4 Auto Tags (自动标签) 业务规则
*   **自适应上下文感知打标**：
    *   系统除了把全文送交 LLM，还会特别附加一个额外上下文信息：**该笔记当前的最新自动摘要（Summary Context）**。
*   **用户习惯拟合（Fewshot 少量样本学习）**：
    *   系统会提取当前库中频次最高的前 100 个历史标签作为“已有标签池”限制。
    *   同时，从数据库中提取该用户最近进行的 3 次“手动修改/纠正 AI 标签”的日志作为 **Few-Shot（少样本微调）** 提示词注入 LLM，强制命令 LLM 拟合用户的打标偏好和风格，杜绝生成天马行空的无用标签。

### 6.5 Smart Rename (智能重命名) 交互与文件系统同步逻辑
*   **防覆盖保护机制**：
    *   重命名是非常高危的文件系统操作。如果检测到用户已经对此笔记手动编写了标题（`user_title` 不为空），或者 AI 已经为此笔记生成了标题，系统会**立即跳过并不予更改**，确保用户的主观意志最高优先权。
*   **安全净化（`sanitize_filename`）**：
    *   大模型生成的拟定标题必须经过 Rust 底层严格的文件名安全净化，物理剔除掉类似 `/`、`\`、`.`、`:` 等潜在的目录穿越或系统敏感字符，防止文件名导致磁盘写入崩溃。
*   **防 Watcher 抢跑的文件系统与数据库“事务级更名同步”**：
    *   由于应用挂载了全局文件监视器（File Watcher），如果在更名时，先改磁盘，Watcher 可能会因为高频异步检测到新文件而抢先将其作为“新笔记”写入数据库，导致老笔记的 `ai_metadata` 数据库记录彻底丢失损坏。
    *   **Slash 采用了极其高超的“持锁同步更名”解决方案**：
        1.  **在 DB 连接锁的保护下**，率先在数据库中执行 `rename_note_in_db`。
        2.  **在不释放 DB 连接锁的情况下**，立即调用 `std::fs::rename` 执行物理更名。这会强行让 Watcher 后台线程在尝试更新数据库时被 Connection 锁阻塞挂起。
        3.  物理更名成功后，顺势更新 `notes` 表中的 `title`、`user_title` 等关键字段，并同步刷新 `ai_metadata` 中对应的记录。
        4.  此时释放 DB 连接锁，Watcher 后台线程被唤醒，但发现数据库早已同步完成，完美避开了时序竞争冲突。
        5.  **级联双链重构同步（Wikilinks Update）**：重命名成功后，系统自动在数据库中搜寻所有含有双链引用（如 `[[老文件名]]`）的笔记，调用 `update_wikilinks_on_rename` 算法，在物理文件上直接将它们重写更新为 `[[新标题]]`，保证整个知识网络绝无死链。
        6.  重命名成功后，系统还会自动更新该 Markdown 文件本身的 Frontmatter 元数据中的 `title` 字段，实现物理级统一。

### 6.6 Smart Archive (智能归档/分类) 业务规则
智能分类能够将未整理的零散笔记（Inbox 笔记）精准推荐移动到最佳的局部主题目录下，避免结构混乱。
*   **文件夹语义特征向量重建（`FolderEmbedding`）**：
    *   为了将文件夹进行向量化，系统会为每个文件夹动态抽取统计学 Profile：包含该文件夹下所有笔记的高频标签（Top-10）、最新的 5 篇笔记标题、笔记的总数等。
    *   组合成结构化文本后，输入向量模型，生成表征该文件夹语义的 **`FolderEmbedding` 文件夹特征向量**，存储在本地数据库中。
*   **未完成任务（Pending Tasks）感知硬路由过滤**：
    *   **大写特权规则：在智能分类前，系统会自动检索该笔记下的所有待办任务。若发现其中含有未完成（`status = 'pending'`）的待办项，系统将强行实施硬路由拦截！**
    *   此拦截将强制剔除掉任何带有“已归档/完成/参考归类”字样的文件夹候选，只允许推荐移动至活跃的执行期工作夹，防止有待办任务的笔记被过早冷冻。
*   **LLM 精准研判与大模型自动二轮修复机制（LLM Repair）**：
    *   结合向量计算相似度最高的 Top-5 文件夹，将笔记 Profile 递交给 LLM 研判，LLM 将输出 `Select`（选择已有）或 `Create`（大类下新建子文件夹，如在 `03_Resources` 下创建 `DeepLearning`）。
    *   如果大模型返回的 JSON 数据结构由于各种网络波动、本地推理截断导致损坏，系统会立即启动**大模型自动二轮修复机制（LLM Repair）**：向 LLM 再次发送请求并提供错误片段令其修复。若修复依然失败，则自动启动**安全向量降级兜底**：直接抛弃大模型决策，改用纯向量相似度排名前三 of 文件夹作为推荐；若依然没有候选，则强行推荐放入通用资源根目录 `03_Resources`。
*   **语义近似防碎片化去重规则（Semantic Deduplication）**：
    *   若大模型提案了 `Create`（如新建 `03_Resources/React-Docs`），系统会先去现存的文件夹列表中计算名称语义相似度。
    *   若发现有一个极度近似的文件夹已存在（如 `03_Resources/React`，相似度评分超过 50%），**系统会无情地强行重写 LLM 的决策，将其改为 `Select` 现存目录，并在原因中特别注明 `[已有近似目录，原提案: React-Docs]`**。这极具前瞻性地遏制了用户知识目录的发散和碎片化。
*   **移动物理与 11 张关联表级联迁移**：
    *   用户接受分类推荐并点击确认后，系统自动物理移动文件。
    *   在数据库事务中，**系统会级联更新多达 11 张关联数据库表**（包含 notes、links、tasks、embeddings_v2 等），确保数据关系的绝对完整。
    *   移动完成后，系统会把源文件夹与目标文件夹同时标记为 `is_dirty = true`，以便在 5 分钟后的增量后台扫描中重新生成它们的特征向量，实现高精自适应。

### 6.7 Similar Notes (相似笔记) 业务规则
用于在不显式建立双链的情况下，通过强大的混血杂交算法发掘隐藏在知识库深处的 semantic 跨界强相关。
*   **图谱与同级目录自动排除机制**：
    *   **物理去重**：自动排除已经手动创建了双链指向的笔记，以及已被用户点击“忽略推荐”拉入黑名单（`ghost_link_blacklist`）的笔记对。
    *   **层级拦截**：
        *   **同级兄弟节点排除**：在同一个物理文件夹下的笔记默认不予推荐（因为物理位置一致代表其已被归类在一起，无跨界推荐价值）。但有一个特权豁免：**只要源笔记或目标笔记中至少有一个驻扎在 `00_Inbox` 中，同级兄弟排除即可被豁免**。
        *   **直系父子节点排除**：父文件夹的引导笔记与直系子目录的笔记，由于具有直接物理包含关系，属于直系上下级，系统**任何时候都绝对不予推荐**（即便是在 `00_Inbox` 中也不予推荐）。
*   **三维一体混血杂交评分公式 (Hybrid Semantic Score)**：
    系统创新地融合了纯向量计算、标签网络和双链拓扑图三种维度的相似度算法，以加权混合得分作为最终推荐度：
    $$\text{HybridScore} = 0.5 \times \text{Sim}_{\text{Vector}} + 0.3 \times \text{Sim}_{\text{Tag}} + 0.2 \times \text{Sim}_{\text{Graph}}$$
    1.  **向量语义相似度 ($\text{Sim}_{\text{Vector}}$ - 权重 50%)**：计算两笔记 Embedding（优先取 `note_profile` 笔记画像特征向量，次选 `paragraph` 向量）的余弦相似度。
    2.  **标签相似度 ($\text{Sim}_{\text{Tag}}$ - 权重 30%)**：使用 Jaccard 算法计算两笔记所有人工及 AI 标签集合的交并比。
    3.  **图拓扑相似度 ($\text{Sim}_{\text{Graph}}$ - 权重 20%)**：计算两笔记在双链图谱中所有相连的“共同邻居笔记”节点集合的交并比，物理位置越靠近得分越高。
*   **文档篇幅自适应动态起算阈值（Dynamic Threshold）**：
    为了抹除长篇幅与短文本在相似度计算上的统计学偏差，系统会对基础过滤阈值（`base_threshold`，如 0.60）进行针对两端大小惩罚/宽免自适应调整（`calculate_dynamic_threshold`）：
    *   若源文档少于 200 字，阈值**上提 $0.10$**（变严格，防无语义噪声干扰）；若源文档大于 10000 字，阈值**下调 $0.05$**（变宽松，方便长文发掘隐藏连接）。
    *   若目标文档少于 200 字，阈值**上提 $0.05$**。
    *   最终自适应后的动态阈值将被强行 Clamp 在 `[0.30, 0.95]` 区间内，得分大于此动态阈值方可被判定为“相似笔记”向用户引荐。
