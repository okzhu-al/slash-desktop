# 当前编辑会话句柄规则

## 目标

当前编辑会话句柄用于把一次连续编辑定义为一个明确的生命周期。它同时服务三件事：

- 判断谁当前有权写入笔记。
- 决定历史版本是否合并为同一条记录。
- 保护同步过程，避免无编辑权页面或旧正文覆盖他人修改。

这套规则适用于个人笔记、团队 Solo 笔记、团队 Collab 笔记。不同模式只改变句柄归属权，不改变生命周期语义。

## 核心概念

### 编辑会话

编辑会话从用户产生明确编辑意图开始，到用户连续空闲超过 60 秒结束。

明确编辑意图包括：

- 点击正文编辑区域准备编辑。
- 输入内容。
- 粘贴内容。
- 拖放内容。

不应被视为编辑意图：

- 打开笔记。
- 程序化加载正文。
- 光标恢复。
- 选择变化。
- 同步 pull 后刷新编辑器内容。
- 历史版本预览。

### 编辑会话句柄

编辑会话句柄是一次编辑生命周期的稳定标识。客户端在编辑会话开始时生成，并在同一会话内复用。

同一会话内多次保存、同步、推送，都应携带同一个句柄。超过 60 秒 idle 后再次编辑，应生成新句柄。

### 编辑权

编辑权决定当前客户端是否允许把正文写入本地文件并 push 到服务端。

编辑权不是“页面是否打开”，也不是“用户是否在团队中”，而是当前模式下的句柄归属结果。

## 模式规则

### 个人笔记

个人笔记的编辑权永远属于当前本地 Editor。

规则：

- 打开个人笔记不会创建编辑会话。
- 第一次真实编辑创建编辑会话句柄。
- 60 秒 idle 内的多次编辑属于同一会话。
- 同一会话内产生的历史版本合并为一条记录。
- 60 秒 idle 后再次编辑生成新历史版本。

### 团队 Solo 笔记

团队 Solo 笔记的编辑权永远属于笔记 Editor。

规则：

- 非 Editor 用户只能只读查看。
- Admin 或 Owner 不应绕过 Solo 编辑权直接编辑活跃成员的 Solo 笔记。
- 原 Editor 离开团队后的孤儿继承规则可以转移 Editor 归属。
- Editor 的编辑生命周期与个人笔记一致。
- 非 Editor 客户端不得通过同步 push 覆盖 Solo 笔记。

### 团队 Collab 笔记

团队 Collab 笔记的编辑权属于当前抢到悲观锁的用户。

规则：

- 打开 Collab 笔记默认是 idle + 只读。
- 只有明确编辑意图才尝试抢锁。
- 程序化 update 只能续期已有锁，不能从 idle 抢锁。
- 抢锁成功后，该用户获得编辑权。
- 抢锁失败时保持只读，并显示当前锁持有人。
- 锁持有人连续 idle 超过 60 秒后释放锁。
- 其他用户可在锁释放后通过明确编辑意图重新抢锁。

服务端必须校验 Collab 写入：

- Collab 模式下，用户编辑 push 必须来自当前有效锁持有人。
- 无锁、锁过期、锁属于其他用户时，服务端必须拒绝正文写入。
- 资产同步不应被正文锁误伤，但正文 `.md` 用户编辑必须受锁保护。

## 历史版本合并规则

历史版本以编辑会话句柄作为合并边界。

同一条历史版本必须满足：

- 同一个 vault。
- 同一个文件身份或同一路径。
- 同一个作者。
- 同一个 `edit_session_id`。

满足以上条件时，后续保存更新同一条历史版本内容、hash、更新时间。

以下情况必须创建新历史版本：

- 第一次创建笔记。
- `edit_session_id` 不同。
- 作者不同。
- 文件不同。
- 超过 60 秒 idle 后再次编辑。
- 由恢复、重命名、状态切换等语义操作产生的独立版本。

服务端不应再使用固定 5 分钟窗口合并历史版本。合并边界以客户端传入的编辑会话句柄为准。

## 同步规则

同步通道是正文的权威数据通道。历史版本不是实时正文通道。

规则：

- Editor 正在编辑时，同步 pull 不得覆盖当前编辑中的正文。
- 失去编辑权或进入只读状态时，客户端必须取消 pending save，清除 dirty 标记，并移出 editing path。
- before sync 只能 flush 当前有编辑权的内容。
- 只读页面不得在退出、切换笔记、同步前 flush 旧正文。
- 服务端必须拒绝无编辑权客户端的正文 push。

## 历史版本预览规则

历史版本预览表达的是“笔记历史版本”，不是“远端实时内容”。

规则：

- 预览列表应与本地同步状态保持一致。
- 用户不能通过已存在的历史版本条目提前读取尚未同步到本地的最新正文。
- 如果历史版本内容 hash 与列表条目 hash 不一致，预览应提示等待同步完成，而不是展示内容。
- 快照预览可以省略重型媒体渲染，例如视频。
- 省略渲染不改变原始历史版本内容。
- 另存为笔记、恢复历史版本时，仍应使用原始内容和原始媒体引用。

## 锁状态展示

Collab 锁状态只表达当前编辑权状态。

- `idle`：无人编辑，不显示编辑中状态。
- `loading`：正在尝试获取锁。
- `acquired`：当前用户编辑中。
- `locked_by_other`：其他用户编辑中。
- `offline`：协作通道不可用，禁止编辑。

用户打开笔记但没有编辑意图时，不应显示“当前用户编辑中”。

## 断线与恢复

客户端应在 access token 临期前主动刷新 token。

规则：

- Collab 轮询、锁请求、已读游标更新都应使用新鲜 token。
- 401 后可以刷新并重试一次。
- refresh token 缺失或失效时，客户端进入认证过期状态。
- 认证过期不应被误报为普通网络断联。
- Windows 与 macOS 都必须通过系统凭据库保存 refresh token。

## 测试用例

### 打开不抢锁

1. User A 打开 Collab 笔记，不点击正文。
2. User B 观察锁状态。
3. 预期：不显示 User A 编辑中。

### 明确编辑才抢锁

1. User A 打开 Collab 笔记。
2. User A 点击正文或输入内容。
3. 预期：User A 获得锁，显示编辑中。

### 程序化刷新不抢锁

1. User A 打开 Collab 笔记但不编辑。
2. 服务端同步 pull 触发正文刷新。
3. 预期：User A 不因程序化 update 抢锁。

### 非锁持有人不能覆盖

1. User B 获得 Collab 锁并编辑。
2. User A 保持同一笔记页面打开但没有锁。
3. User A 触发同步或切换页面。
4. 预期：User A 不会 flush 旧正文；即使发生 push，服务端也拒绝。

### 60 秒内合并历史版本

1. 用户连续编辑同一笔记，多次保存间隔均小于 60 秒。
2. 查看历史版本。
3. 预期：合并为同一条历史版本。

### 60 秒后创建新历史版本

1. 用户编辑同一笔记。
2. 空闲超过 60 秒。
3. 再次编辑并同步。
4. 预期：创建新历史版本。

### 预览不绕过同步

1. User A 修改 Collab 笔记。
2. User B 尚未 pull 到正文。
3. User B 打开历史版本预览。
4. 预期：不能通过旧条目看到尚未同步到本地的最新正文。

### 视频预览省略但可恢复

1. 历史版本内容包含视频引用。
2. 打开历史版本预览。
3. 预期：预览显示视频省略提示，不加载视频。
4. 恢复或另存为笔记。
5. 预期：原始视频引用仍然保留。

## 发布检查

发布前必须确认：

- 个人笔记编辑历史按 60 秒 idle 合并。
- 团队 Solo 非 Editor 无法写入。
- 团队 Collab 打开不抢锁。
- 团队 Collab 明确编辑才抢锁。
- 非锁持有人 push 被服务端拒绝。
- 同步 pull 不覆盖当前编辑者内容。
- 历史版本预览不泄露未同步正文。
- token 刷新在 Windows 与 macOS 都可用。
