# Slash Desktop 基础功能后置 PRD 与业务规则说明书

## 1. 文档概述

本页为 Slash Desktop 客户端基础核心功能的后置业务规则说明书（后置 PRD）。通过逆向工程分析客户端前后端底层实现代码，将隐藏在 Rust 核心命令层（Tauri Commands）、本地 SQLite 数据库迁移机制、文件监控器（Watcher）与前端 Zustand 存储及 React 钩子中的实际业务规则提取为可追溯的标准化产品定义。

本说明书侧重阐述目前系统已实现功能的实际逻辑、状态机边界、限制条件及同步策略。

## 2. 仓库初始化与结构定义

### 2.1 PARA 物理目录的强制定义与自动化重建
当用户打开或新建一个本地仓库（Vault）时，系统会在前端（`ParaService`）与 Rust 后端（`db.rs` / `VaultConnection`）双重并发触发防御性检查，强制规范并创建标准的 **PARA（Projects, Areas, Resources, Archives）** 笔记管理物理目录结构。

*   **强制目录定义**：
    系统在仓库根目录下强制初始化并维持以下物理文件夹：
    *   `00_Inbox`：用于存放所有新建但未归类的零散笔记。
    *   `01_Projects`：存放当前正在推进、有明确目标和截止日期的项目。
    *   `02_Areas`：存放长期需要投入精力并维持高标准的领域。
    *   `03_Resources`：存放未来可能用到的参考资料或兴趣主题。
    *   `04_Archives`：存放已完成或失效的归档内容。
    *   `assets`：存放所有笔记内插入的本地多媒体附件（如图片、音频等）。
*   **前端/后端双重防御性机制**：
    *   **前端验证**：在 App.tsx 的 `handleVaultOpened` 流程中，会首先调用 `ensureParaStructure(vaultPath)` 在磁盘中以异步非阻塞方式确保上述文件夹存在。
    *   **后端补充**：在 Rust 核心初始化命令 `init_db` 内，系统会再次调用 `ensure_para_structure(&vault_dir)` 进行补充防御。若上述任一物理文件夹缺失，系统会自动在磁盘上建立对应目录，保证文件路径可用性，防止后续笔记写入时因父级目录缺失发生 I/O 报错。

### 2.2 本地 SQLite 数据库初始化与版本迁移
本地客户端采用轻量级 SQLite 数据库（存放在仓库根目录的 `.slash/slash.db`）管理非结构化的 Markdown 文件元数据。

*   **独占式多路并发管理**：
    *   数据库连接被包装在 Rust 后端的全局 `DbState` （内部通过 `Mutex` 互斥锁保护的 `SqliteConnection`）中，确保同一个进程中对数据库的操作是线程安全的。
*   **Schema 演进与迁移规则（V1 - V34）**：
    数据库模式通过统一的迁移器 `migrations.rs` 进行渐进式构建。主要业务版本演进规则包括：
    *   **V1-V4（基础物理映射与全文搜索）**：
        创建 `notes`、`links` 表，并定义 `notes_fts` 全文搜索虚拟表，利用 SQLite 内建 FTS 引擎实现毫秒级笔记全文检索。
    *   **V8-V10（任务与级联状态机）**：
        引入 `tasks` 表，用于抽取笔记中的待办事项（Todo Items）。
    *   **V12-V19（全文检索同步与多维元数据）**：
        通过创建 SQL 触发器（Triggers），在 `notes` 表发生 `INSERT`、`DELETE` 或 `UPDATE` 时，自动级联将内容同步更新至 `notes_fts` 表，确保搜索数据绝对一致。同时引入 `ai_metadata` 和 `embeddings` 元数据表。
    *   **V23-V34（协作与向量化索引升级）**：
        引入 `ai_skill_state` 状态机记录、升级 `embeddings_v2` 向量表（增加 `chunk_index` 及 `hash` 段，支持段落级语义分割与增量清洗）。

## 3. 笔记与目录的 CRUD 与命名冲突规则

### 3.1 命名限制与特殊字符过滤
为确保笔记和文件夹在跨平台（Windows/macOS/Linux）文件系统下的物理兼容性，系统在命名上实施严格的限制：

*   **空命名拦截**：
    前端 `confirmRename` 强行拦截空输入。任何去除首尾空格后为空字串的命名操作（`!newNameRaw.trim()`）均会被静默丢弃，不发起任何 I/O 或数据库写入。
*   **非法特殊字符物理过滤**：
    前端通过统一的工具方法 `sanitizeFilename` 过滤文件名：
    *   **过滤正则**：`/[<>:"/\\|?*]/g`
    *   **过滤行为**：将 Windows 文件系统保留字与 Unix 目录分割符（包括 `<`、`>`、`:`、`"`、`/`、`\`、`|`、`?`、`*`）直接替换为空字符（即无声抹除），并自动执行首尾 `.trim()` 去除冗余空格。

### 3.2 新建文件/目录时的重名自增计数器机制
在新建笔记或文件夹时，为避免与已有文件发生物理碰撞，系统在上层实现了重名自增保护：

*   **创建笔记的防撞逻辑**：
    *   默认新建名称为 `未命名笔记` (或本地化后的 `"New Note"`)。
    *   前端通过一个自增计数器循环（上限 100 次）不断调用底层 Repository 的 `createNote` 接口。
    *   若发生物理碰撞导致底层抛出 `Note already exists` 异常，计数器累加，重新拼接为 `未命名笔记2`、`未命名笔记3`，直至写入成功。
*   **创建文件夹的防撞逻辑**：
    *   逻辑与新建笔记一致，默认名称为 `新建文件夹` (或本地化后的 `"New Folder"`)，若重名则自动递增后缀，上限 100 次。

### 3.3 新建笔记初始化前置规则
系统在物理创建笔记文件时，除了写入空白文本，还会自动格式化 YAML 前置信息（frontmatter），这是团队同步和 AI 逻辑正常运作的关键前哨：

1.  **全局 UUID（`slash_id`）前置生成**：
    在文件写入磁盘前，系统使用前端 `crypto.randomUUID()` 生成唯一的 UUID，写入 frontmatter 的 `slash_id` 键中。这确保了首次多端同步（push sync）时，云端能根据此 UUID生成一致的快照，规避后续同步过程中因缺失 ID 产生重复创建的异常。
2.  **默认编辑属性初始化**：
    *   若用户已登录，注入当前用户的 `displayName` 到 `editor` 字段。
    *   初始化 `contributors` 数组为 `[displayName]`。
    *   设置协同模式属性 `doc_status` 默认为 `'solo'`（仅 Editor 可编辑）。
3.  **同名历史 AI 索引深度清洗**：
    在写入物理文件前，系统强制调用 Rust 命令 `purge_stale_note_data`，将该文件路径下所有历史遗留的 AI 元数据、嵌入向量等在数据库中物理清除，确保老同名笔记的历史痕迹不会污染新笔记本尊。

### 3.4 删除笔记时的 Tab 关闭等待与防幽灵复活机制
物理删除笔记时，极易因高频自动保存（Auto-Save）发生“被删文件幽灵复活”现象。为此，系统制定了严密的防复活时序：

1.  **Tabs 过滤与强制关闭**：
    前端首先过滤出所有匹配被删文件/文件夹路径的 Tab ID（包含子孙路径），调用 `closeTab` 强制将对应编辑页面卸载（Unmount）。
2.  **250ms 自动保存延迟等待**：
    **【关键时序】** 关闭编辑页后，前端采用 `await new Promise(resolve => setTimeout(resolve, 250))` 强制挂起 250 毫秒。此举旨在确保因 Tab 卸载被动触发的底层最后一次 `save_note` 物理保存逻辑能够在磁盘上完全写完。
3.  **彻底物理删除**：
    待可能并发的写入事务全部沉淀后，再调用后端 `deleteNote` 执行物理删除并同步更新 Zustand 内存树（`removeNode`），这确保了文件不会在被删除后又被缓存中的脏数据反向写盘而“复活”。
4.  **垃圾箱托管（move_to_trash）**：
    底层的物理删除并非直接抹除（`std::fs::remove_file`），而是优先调用系统级别的垃圾箱托管。在 Windows/macOS 上将文件安全移入回收站（Trash）。如果移入回收站命令失败，系统才会降级为物理 `remove`。

### 3.5 移动与重命名时的前后端同步操作时序
对笔记重命名或移动位置，是一系列复杂的数据库、文件系统、与云端协同逻辑的同步传导过程。

*   **重名首道防线拦截**：
    在重命名笔记时，前端先调用后端的 `check_note_exists` 在 SQLite 数据库查重。若已存在重名笔记，直接弹出警告阻止改名。
*   **不区分大小写文件系统的大小写修改（Case-Only Rename）**：
    由于 macOS/Windows 的物理文件系统默认不区分大小写，直接对文件重命名为大小写变体（例如 `A.md` 改为 `a.md`）会报错或失效。系统采用**双步改名策略**：
    1.  第一步：先将物理文件重命名为唯一的临时名称（如 `tmp_rename_时间戳.md`）。
    2.  第二步：再将临时名称文件重命名为目标字母大小写名称。
*   **【重中之重】物理改名前数据库迁移先导（Database Migration First）**：
    对文件系统执行物理重命名前，**必须先调用后端 `rename_note_in_db` 对数据库中的记录做位置迁移**。
    *   **业务目的**：当发生重命名时，底层的磁盘文件监听器（Watcher）会几乎实时捕获到文件变动。如果先物理改名，Watcher 会立即认为“旧文件被删除，新文件被创建”，进而触发级联清理规则，抹去该笔记原有的所有任务关系、语义向量和 AI 链接。
    *   **设计解法**：在物理改名前，先在 SQLite 事务中完成路径重定向。当物理文件重命名被 Watcher 捕获时，它在数据库中已能成功匹配新路径，不会误判为“删除+新建”，从而实现了完美的热更新。
*   **同步修改文件 YAML Title**：
    文件物理重命名成功后，系统会再次读取文件，将 frontmatter 中的 `title` 更新为对应的新文件名，保证文件内容与文件名展示相一致。
*   **团队协同目录重命名的回滚机制**：
    若是映射在团队协作空间下的目录改名，系统会同步更新 `team_path_mappings.json` 本地映射，并调用 `teamService.renameDirectory` 向云端发起改名。若因权限不足被云端拒绝，前端会执行物理回滚：调用物理重命名改回原名并刷新，防止两端状态不一致。

## 4. 级联数据维护与资产清理规则

### 4.1 物理删除/重定向时的数据库元数据级联维护
当笔记在物理上被删除或移动时，为防止数据库产生悬空数据或索引垃圾，系统会通过事务和级联规则自动擦除相关元数据：

*   **删除时级联清除范围**：
    在 `db.rs` 中的 `delete_note` 以及物理删除触发器中，系统无条件同步销毁：
    *   `notes` 主表记录。
    *   `links` 关联表中的出链、入链记录（避免悬空图谱链接）。
    *   `tasks` 表中抽取自该笔记的所有待办任务。
    *   `embeddings_v2` 表中的所有向量特征段，避免废弃向量在 AI 检索中被拉出。
    *   `ai_metadata` 中关于本笔记的实体、总结及相关 AI 属性。
    *   `ai_skill_state` 中的 AI 技能演进状态。
*   **重命名的事务隔离保证**：
    在 `rename_note_in_db` 逻辑中，由于需要修改主键字段（`notes.path`），为防止外键约束在修改过程中由于级联不一致导致修改报错，系统在底层使用 `SAVEPOINT` 显式事务，在事务中临时关闭外键约束检测（`PRAGMA foreign_keys = OFF`），待更新完 `notes.path` 及相关联的 `links.source` / `links.target` / `tasks.note_path` / `ai_metadata.note_path` / `embeddings_v2` 后，重新开启外键约束，保证了多级联数据维护的原子性与完整性。

### 4.2 孤儿资产（附件）的离线引用计数分析与静默清理
用户在编辑 Markdown 时，经常会插入图片附件，但删除或替换这些附件引用时，磁盘上的物理附件（`assets/` 目录下）会成为孤立文件（Orphan Assets），白白占用磁盘空间。

*   **资产变动触发点**：
    *   **触发点 A（编辑变动）**：每次 `saveNote` 时，系统会对比旧文件内容与新文件内容中的资产引用列表。若存在被移除的资产引用，触发清理。
    *   **触发点 B（笔记删除）**：删除笔记时，在执行物理删除前，先提取该文件内的资产引用列表。待文件彻底陨灭后，触发对这些资产引用的清理。
*   **孤儿判定（引用计数扫描）**：
    *   系统提取待检测资产的相对路径（如 `assets/xxx.png`）。
    *   递归遍历整个库下所有的 `.md` 笔记文件（跳过隐藏文件和 `assets` 目录本身）。
    *   读取每一个 Markdown，统计对待检测资产的全局引用次数。
    *   若全局引用次数降为 **0**，则判定该附件为“孤儿资产”；若大于 0，则保留（KEEP）不进行处理。
*   **静默物理清理**：
    *   对判定为 0 引用的孤儿资产，系统会调用 Tauri 命令 `move_to_trash` 将 `assets/xxx.png` 移入系统回收站，从而实现对残留图片附件的静默垃圾回收。

## 5. 仓库切换与身份隔离机制

### 5.1 数据库物理关闭与连接释放
为了防止切换本地仓库时出现脏数据或文件锁定异常，系统设计了严密的数据库生命周期管理机制：

1.  **取消监听与事务取消**：
    当检测到要切换仓库或关闭当前库时，前端 Hook 会首先阻断（Cancel）当前正在执行的任何初始化或扫描流程，同时取消对 `vault:refresh` 的全局事件监听，截断后续的消息投递。
2.  **物理释放连接（close_db）**：
    调用后端的 `close_db` 命令。后端在 Rust `DbState` 的 Mutex 互斥锁保护下，重置已打开的 SQLite 连接。
    *   **业务效果**：这使得 SQLite 文件上的独占锁被完全释放。这样在进行仓库切换、物理移动或删除旧仓库的文件夹时，不会因为文件被 Rust 进程锁定而出现 “Permission Denied” 或 “File in use” 等报错。

### 5.2 跨仓库（Vault）切换的零信任登出防护规则
为了防范会话污染、保护个人与企业团队间的数据隐私，系统设置了强力的**零信任防线（Zero-Trust Guard）**：

*   **跨 Vault 切换强制登出**：
    在 App.tsx 的 `handleVaultOpened` 方法中：
    ```typescript
    if (vaultPath && vaultPath !== path) {
      console.log('[App] Explicit vault switch detected. Enforcing total logout to prevent session crossover.');
      useSessionStore.getState().clearAll();
      syncService.clearConfig();
    }
    ```
    *   **规则逻辑**：若用户当前已经打开了一个仓库（`vaultPath` 非空），且接下来选择要切换到的仓库路径与当前路径不一致时（即：显式跨仓库切换）。系统会**强制清空所有全局身份凭证**。
    *   **具体动作**：
        1.  调用 `useSessionStore.getState().clearAll()`，擦除内存与 LocalStorage 中的用户登录状态、用户名、访问令牌等所有全局身份凭证。
        2.  调用 `syncService.clearConfig()`，清理同步服务器终结点和已加载的配置 Token。
    *   **业务防范目的**：确保了每个本地仓库与其云端协作空间保持物理隔离（"1 Vault = 1 Team"）。绝不允许在一个仓库中配置的登录凭证（如公司内部敏感项目的 Token）残留到另一个仓库（如个人仓库）中使用，杜绝了会话越权同步和数据交叉泄露的重大安全隐患。
