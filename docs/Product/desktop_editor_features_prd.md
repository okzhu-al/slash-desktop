# Slash 编辑器编辑功能后置 PRD 业务规则文档 (Batch 2 ~ Batch 2.8)

本 PRD 详细阐述 Slash 桌面端编辑器在富文本格式化继承、混合列表层级嵌套退格、表格节点构建控制、行内代码与公式块排版，以及双链 [[WikiLink]] 级联落盘的一系列后置产品级业务规则与交互设计规范。

---

## 1. 文本排版与格式继承规则

### 1.1 样式继承阻断与连续性规则 (keepOnSplit)
* **业务场景**：用户在已经应用了行内样式（如加粗 **Bold**、斜体 *Italic*、下划线 Underline、删除线 ~~Strike~~、高亮 ==Highlight==）的文本行末尾，按下 `Enter` 键进行折行。或者在已有样式文本的中间/边缘进行插入输入。
* **业务规则**：
  * **换行格式阻断**：当光标处于行内格式化文本的**绝对末尾**（即样式边界的右侧边缘）按下回车时，新生成的段落**严禁继承**上一行的任何行内样式标记。新起一行输入的所有新字符必须默认归一化为普通未格式化文本（Bug 4 修复）。这保证了用户在结束当前样式段落换行后，能够流畅开始书写普通正文，无需手动关闭之前的一系列格式。
  * **中间输入格式连续**：当光标处于格式化文本的**内部**（例如，`加|粗` 的中间位置）或样式边界的**左侧边缘**时，用户的插入输入或中文输入法空格确认，**必须保持**该行内样式的连续性，不得发生非受控的样式丢失（Bug 6 & Bug 8 修复）。
  * **物理层级防漏**：在序列化为 Markdown 时，行内格式的闭合标记（如 `**`、`*`、`==`）必须精确包裹对应文本，不得产生空标记对（如 `****` 或 `====`），以免造成反解析时的 AST 树异常。

### 1.2 快捷格式退格还原规则 (Backspace Revert)
* **业务场景**：用户在紧贴着行内样式（如加粗 `**`、行内代码 `` ` ``、高亮 `==` 等）的最右侧边缘处按下 `Backspace` 键，试图修改或撤销该格式。
* **业务规则**：
  * **一步到位语法还原**：为了提升高级写作者的编辑效率，当按下退格键时，编辑器**严禁**将加粗文本或行内代码视作不可分割的黑盒整体直接删除，也**严禁**只删除文字而留下残留格式占位符。系统必须立即**剥离该文本的行内样式格式**，并当场将其还原为带有所属 Markdown 标记语法字符的普通纯文本形式（例如，`**bold|` 按下退格键立即变为 `**bold|` 的普通文字形态；`` `code` `` 还原为 `` `code `` 并退出代码 mark 状态）（Bug 5, Bug 21, Bug 22 优化）。
  * **光标位置与状态重置**：格式还原后，光标必须精准保持在纯文本的末尾。同时，必须显式清除编辑器的 StoredMarks 状态，确保紧随其后输入的新字符不会被污染，继续保持为纯普通文本输入。

## 2. 混合嵌套列表与退格交互规则

### 2.1 顶级异构列表的物理区隔规则
* **业务场景**：编辑器中相邻的两个顶级（非嵌套）列表节点属于不同的列表类型（例如：一级有序列表 `1. ` 后面紧跟着一个一级任务列表 `- [ ] `）。
* **业务规则**：
  * **异构粘连预防**：当相邻列表类型不一致时，在编辑器向 Markdown 物理文件进行序列化写入时，**必须强制输出两个换行符 `\n\n`** 进行物理区隔（即在两个列表之间留有一个绝对空行）（Bug 11 优化）。
  * **产品防错逻辑**：如果省略该物理空行，在下次重新载入该文件时，Markdown 解析器（markdown-it / mdast）会将这两个连续的顶级列表归入同一个父级列表容器中，并因类型转换纠错机制导致无序列表被误解析为带有空 checkbox 的任务列表，或者有序列表编号被强行覆盖。

### 2.2 嵌套列表垂直行距一致性规则
* **业务场景**：在第一级任务列表项内部嵌套第二级列表（如：有序列表、无序列表或子级任务列表）。
* **业务规则**：
  * **视觉层级高度一致**：无论列表嵌套多深，二级及以上嵌套列表的顶部（`margin-top`）在 CSS 渲染层必须表现出恒定且一致的 `0.35rem` 外边距（Bug 12 修复）。
  * **覆盖多层包裹容器**：在富文本排版中，嵌套列表由 React 包装渲染组件承载，导致 DOM 结构中包含多层冗余容器。样式系统必须使用强力的后代选择器并强制重置（`!important`），消除底层组件的默认 margin 溢出，确保嵌套段落的垂直缩进间距具有完美一致的呼吸排版感。

### 2.3 异构类型退格提升与跨界合并规则
* **业务场景**：光标处于二级的空列表项行首（即该子级列表无任何文本内容），用户按下 `Backspace` 键。
* **业务规则**：
  * **第一阶段：同异构转换提升**：
    * 若二级列表项的类型与父级一级列表的类型**不同**（例如：一级为有序列表项，二级为空任务列表项），按下退格键时，系统必须将其安全地提升到一级列表层级，并**原地将其转换为一级列表的类型**（即子级任务项转换为一级有序列表项，与其原 parent同级），保留其所有的子节点和后续行高，严防文档结构断裂崩溃（Bug 13 修复）。
    * 若二级列表项类型与父级**相同**，则执行原生提升（即直接提升为上一级的同类型列表项）。
  * **第二阶段：顶级空段落跨界合并**：
    * 当列表项通过多次退格彻底退化为文档第一层级的顶级空段落（`depth = 1`）时。
    * 此时，若该空段落的前驱兄弟节点是一个列表（无序/有序/任务列表），用户在此空段落行首再次按下退格键，系统必须自动拦截该事件。
    * **产品防崩溃合并规则**：严禁执行 ProseMirror 默认的将空段落强行塞入前驱列表尾部的合并行为（这会导致列表结构和光标混乱）。系统必须在**单次原子事务**中彻底删去当前的顶级空段落，并将光标精准定位并聚焦至前驱列表中最深分支的最后一个段落的物理文本末尾，实现无缝平滑回退，给用户以极其符合直觉的编辑反馈。

## 3. 表格生成、复制与边界防崩溃控制规则

### 3.1 Markdown 表格快捷匹配创建
* **业务场景**：用户在全新的一行行首输入特定的表格快捷骨架并输入空格。
* **业务规则**：
  * **智能快捷键绘制**：当用户在一行的行首输入 `|| `（双竖线加空格），编辑器必须拦截并自动将其转换为一个 **3行 x 3列 的表格结构**。
  * **光标初聚焦**：表格生成后，第一行自动设定为表头行（`tableHeader`），其余两行为数据行（`tableRow`）。单元格内部默认初始化为一个空的段落，并且编辑器光标必须**自动聚焦并定位到表格的第一个单元格（第 1 行第 1 列）内部**，使用户能够即时开始打字输入，无需使用鼠标二次点击。

### 3.2 表格内容纯文本复制规整
* **业务场景**：用户在表格节点中选中部分区域（或者全部表格）并执行复制（Cmd+C / Ctrl+C）。
* **业务规则**：
  * **网格格式化**：系统监听编辑器的 Clipboard 复制事件。若检测到复制内容中存在表格节点（`table`），在向系统剪贴板中写入 `text/plain` 纯文本格式数据时，必须将其规整转换为：**列与列之间使用制表符 `\t` 分割，行与行之间使用换行符 `\n` 分割**的纯文本网格形式（Bug 18 修复）。
  * **外部粘贴支持**：规整后的纯文本必须彻底剥离 Markdown 表格的物理画线符号（如 `|` 和 `---|`）。这保证了用户将 Slash 中复制的表格粘贴到 Microsoft Excel、Google Sheets、Numbers 或外部普通文本文档中时，能够自动保持完美的行列网格单元对齐。

### 3.3 单元格硬换行 `<br>` 规则
* **业务规则**：
  * 在表格单元格中，如果用户通过快捷键（Shift+Enter）在单元格内部换行，这属于“硬换行”。
  * 在向物理磁盘保存 Markdown 格式时，必须将其序列化为 `<br>` 物理标签进行转义保存。
  * 当重新载入笔记并解析 Markdown 时，必须把所有的 `<br>` 标签重新还原为 ProseMirror 的 `hardBreak` 节点，**绝对不允许**将 `<br>` 作为字面量纯文本展示在单元格内部（Bug 17 修复），以保障排版界面的整洁度。

### 3.4 表格边界操作防白屏崩溃规则
* **业务规则**：
  * 当用户频繁操作表格上方的列手柄、左侧的行手柄，或者使用右键菜单删除最左/最右列、最上/最下行，或者在表格边缘执行行列拖拽时，编辑器底层状态必须拥有严格的边界过滤。
  * 所有的行列索引定位函数在通过 ProseMirror 的 Transaction 更新时，必须进行有效的文档边界裁剪（如 `Math.min(calculatedPos, doc.content.size)`），并在发生边界越界时温和捕获，**绝对不允许**在用户界面抛出 `RangeError` 导致编辑器白屏或数据丢失（Bug 20b 修复）。

## 4. 公式块与引用块交互边界规则

### 4.1 公式块多行隔离与区分规则
* **业务场景**：用户在行首键入美元符号或者使用公式命令。
* **业务规则**：
  * **块级与行内公式物理物理隔离**：
    * **块级公式 (mathBlock)**：当在一行的行首键入 `$$` 且输入空格，或输入 `$$formula$$` 并按回车时，必须将其转化为独立的块级公式。它必须拥有专属的多行 LaTeX 源码编写卡片，并在正中以 display 模式（大公式模式）渲染。
    * **行内公式 (math)**：通过单美元符号 `$formula$` 包裹的公式，必须渲染为与其所在的段落行内文字混排的小公式。
  * 在 Markdown 序列化时，块级公式必须强制保存为单独一行的 `$$\nformula\n$$` 格式，行内公式保存为 `$formula$` 格式。解析时必须精确区分这两种节点，禁止将多行块公式错配成行内公式包裹（Bug 23 修复）。

### 4.2 引用块单行占位及光标定位规则
* **业务场景**：用户在空白段落行首输入 `> ` 并键入空格，或者通过斜杠命令启动引用块（blockquote）。
* **业务规则**：
  * **单行物理引号初始化**：新建的引用块必须初始化为只占 1 行的结构（即包含单个独立的空段落）。
  * **引号分布与无占位符**：前双引号 `“` 和后双引号 `”` 完全由 CSS 伪元素自动渲染在唯一空段落的首尾。在这个段落中，必须禁用普通空行的占位符提示文字，禁用 Tailwind 默认生成的装饰性双引号（Bug 14 优化）。
  * **光标定位**：初始化完成后，光标必须自动放置在唯一的空段落行首（即中间位置），空引用块在视觉上应呈现为 1 行的 `｜“|”` 结构（`｜` 为引用块左侧竖边框，`“` 为前引号，`”` 为后引号，光标闪烁在正中间）。用户键入内容时，字符应在同一行内向两侧推开前、后引号（如 `“输入文字”`），完全消除物理引号重叠及输入法重入冲突。

## 5. 双链 (WikiLink) 数据结构与落盘校验规则

### 5.1 双链文本提取与别名映射规则
* **双链解析**：后端扫描器扫描 markdown content，利用正则表达式 `\[\[([^\]]+)\]\]` 匹配显式双链。
* **物理锚点别名分离**：
  * 解析 `[[Target#Section|Alias]]` 时，目标锚点 `target_anchor` 必须被规整为 `[[Target#Section]]`（剔除别名 `|` 标记，但保留 `#` 章节定位）。
  * 链接标签 `label` 部分保存为别名 `Alias`。若无别名，则 `label` 默认与 `target_anchor` 中去除括号后的目标主题保持一致。
* **代码/公式安全屏蔽**：在从 Markdown 文件提取双链时，扫描器**必须提前清洗**掉 fenced code blocks (```...```)、行内代码 (`` `...` ``)、块级公式 ($$...$$) 以及行内公式 ($...$) 的内容，防止将写在代码注释或公式公式中的 `[[Link]]` 文本误解析为物理双链（链接逻辑安全防卫）。

### 5.2 YAML 元数据关系 (relations) 合并规则
* **显式合并**：
  * 用户可以在笔记的 YAML Frontmatter 中以键值对的形式定义关系，如：
    ```yaml
    relations:
      核心指标: 性能测试
    ```
  * 当后端扫描器在 `scan_file` 时：
    * 解析 relations 中指定的 target（如 `"性能测试"`）。
    * 检查正文显式提取出的 WikiLink 集合。
    * 若在正文中发现了指向目标笔记（如 `[[性能测试#核心指标|测试展示]]`）的 WikiLink，系统必须将其对应的 SQLite `links` 数据表项中的 `label` 属性强制置为该 relations 的键名（即 `label = '核心指标'`）。
    * **孤儿关系清理**：若 relations 中定义了关系，但在正文中没有找到任何匹配的物理 WikiLink，说明该关系悬空，系统必须在扫描入库时从 Markdown 物理文件的 frontmatter 中将该孤立 relation 自动清洗抹除，保持数据的强一致性。

### 5.3 数据库字段物理映射约束与设计
双链扫描入库时，在 `scan_and_upsert` 阶段会写入 SQLite 数据库的 `links` 数据表。各核心字段的物理映射约束定义如下：

| 数据表字段 (Column) | 物理数据类型 (Type) | 约束条件 (Constraints) | 产品与数据业务映射规则说明 |
| :--- | :--- | :--- | :--- |
| **id** | INTEGER | PRIMARY KEY AUTOINCREMENT | 数据库自增主键，保障记录 of 唯一性。 |
| **source_path** | TEXT | NOT NULL | 双链所属源笔记的**相对物理路径**（如 `'00_Inbox/测试文档.md'`）。在外键上与 `notes` 表建立绑定，当源笔记被彻底物理删除时，触发 `ON DELETE CASCADE` 级联删除该记录。 |
| **target_path** | TEXT | NULL | 指向目标笔记的相对物理路径。对于尚未创建目标笔记的双链（即 Ghost Link 悬空死链），**必须写入 NULL**。系统会在图谱渲染或用户点击时进行动态 Resolve。 |
| **target_anchor** | TEXT | NOT NULL | 剔除别名符号的 WikiLink 寻址锚点（如 `'[[性能测试#核心指标]]'`），用于图谱拓扑结构的匹配和章节（Anchor）定位。 |
| **label** | TEXT | NULL | 别名属性。若关联了 Frontmatter 的 relations，这里写入关系键名（如 `'核心指标'`）；若是普通别名双链，则写入别名内容。 |
| **link_type** | TEXT | DEFAULT 'explicit' | 链接类型区分。`explicit` 代表正文显式 WikiLink 链接；`yaml` 代表通过 frontmatter 元数据声明的隐式关系链接。 |
| **created_at** | INTEGER | DEFAULT CURRENT_TIMESTAMP | 链接被写入或更新时的 Unix 秒级物理时间戳，用于追溯双链图谱的版本变迁。 |
