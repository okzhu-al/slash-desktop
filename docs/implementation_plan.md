# Slash 扩张与内测筹备计划

当前 Slash 第一个阶段（核心功能）已基本冻结，目前面临从“开发完毕”走向“内测及商业化”的过渡。为确保内测期间的数据安全和产品体验，我们需要对现有的待办工作进行科学的优先级重排。

## 需求评估与优先级分析

原计划的工作：
1. 生成内测包 (Mac、Windows、Server部署包)
2. 对象存储重构 (防止 Docker 重启丢文件)
3. 客户端热发布 (内测期间避免每次重新安装)
4. iOS/Android 移动端原生应用
5. 商业化变现与订阅制

### 潜在风险评估
- **致命风险 (数据丢失)**：如果对象存储尚未重构便分发 Server 部署包，Docker 一旦重启/迁移，内测用户的文件可能直接丢失，这会引发严重的信任危机。
- **流程痛点 (更新断层)**：热更新/热发布能力必须“预埋”在递交给用户手里的**第一个版本**中。如果首版不包含更新器，之后哪怕配好了热更新，用户也必须再手动走一次重装流程。

---

## 调整后的最优执行路线

基于上述分析，最优路线应遵循 **“底层安全 -> 热更通路 -> 打包分发 -> 端侧拓展 -> 商业变现”** 的顺序进行。建议调整如下：

### 阶段一：内测基础设施改造 (阻断体验风险)
1. **[升优] 对象存储重构 (Object Storage Refactoring)**
   - **目标**: 将附件、图像、离线文件等资源迁移至标准化的对象存储（如内置 MinIO 或兼容 S3 协议的外部服务），让 Docker 容器做到真正无状态 (Stateless)。
   - **原因**: 彻底杜绝 Docker 重启/升级带来的内测数据丢失风险，保障数据这条生命线的最优先级。
2. **[升优] 客户端热发布/自动更新 (Client Hot-Updater)**
   - **目标**: 基于 Tauri v2 Updater 等能力配置客户端全量/增量热更新服务。
   - **原因**: 必须在第一个内测包发出前就包含此机制，确保内测期的高频 Bug 修复能够丝滑下发，降低用户测试阻力。

### 阶段二：打包装盘与内测放量
3. **[延后] 生成安装包与 Server 部署，启动内测**
   - **目标**: 构建 Mac, Windows 的独立安装包与 Server 端的 Docker Compose 部署文档，正式接入内测用户。
   - **原因**: 在基础设施加固（步骤1&2）后进行宣发，才能承接好用户，实现无感平滑修复。

### 阶段三：全端布局与商业验证
4. **iOS/Android 原生移动端应用**
   - **目标**: 利用底层共享架构（考虑 Tauri Mobile 等方案），拓展移动端生态，实现真正的全端漫游。
   - **原因**: 在前端业务与后端存储经过桌面端内测充分“真机验证”变得非常稳定后，移植移动端会事半功倍。
5. **商业化变现与订阅制搭建**
   - **目标**: 搭建 Stripe / 微信 / 支付宝 等支付通路，设计权益隔离体系 (Paywall)。
   - **原因**: 待核心口碑建立和各端稳定后执行，是最具信心且水到渠成的最后一步。
