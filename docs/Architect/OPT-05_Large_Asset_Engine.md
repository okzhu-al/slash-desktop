# OPT-05: 大体量资产引擎重构架构 (Large Asset Engine)

## 核心痛点与问题背景

目前 Slash 系统在处理“文件”时，采用了一套“一刀切”的流转体系。无论是一张 500KB 的插图，还是一个 2GB 的高清视频记录，全都共用一套由前端拉取、Base64 转换、JSON 封装传输的流水线。这导致在现代大存储时代面临四大物理极限碰撞：

1. **前端内存溢出（OOM）**：WebView（V8）内存瓶颈，直接 `readFile` 大文件导致崩溃。
2. **AI 解析阻塞**：大型音视频转录耗时过长，阻塞 UI 导致用户误判卡死。
3. **网络传输脆性**：大文件塞进单次 HTTP JSON `push` 请求，超时断连即全盘失败，且 Base64 增加额外 33% 内存载荷，甚至引发服务端 OOM。
4. **硬编码规则壁垒与不统一**：客户端存在双重标准（硬编码 50MB 团队长传拦截），且各种文件导入入口（拖拽、复制、侧边栏导入）缺少中心化的管控。无法发挥私有部署（本地主机、NAS等）的真实带宽和存储优势。

---

## 全局统一性原则 (Global Unification Principles)

在实施具体重构前，必须确立以下两项不可动摇的统一性铁律：

1. **同构并发（个人与团队共用一套存储规则）**
   * 打破人为壁垒：由于不论是 Personal 还是 Team 空间，底层均由相同的服务器设施承载（同等网络、硬盘），必须彻底废弃代码层面针对空间的区别对待（如：取消个人无限、团队 50M 的分裂设定）。将限制权上浮交由服务器管理员统一裁决。
2. **万流归宗（全渠道输入遵循相同拦截线）**
   * 入口闭环：不论用户是通过**笔记内复制粘贴 (Paste)**、**界面拖拽 (Drag & Drop)** 还是**侧边栏直接导入 (ImportHub)**。所有触发“文件注入”的动作必须经过同一处网关（Gateway）进行 Volume/Stat 拦截。杜绝“拖进去能行，用导入不行”的薛定谔体验。

---

## 四大物理边界分析与交互重构策略

根据“物理约束定交互范式”的第一性原理，将统一的文件流转解构为四个独立的生命周期，并施加针对性的交互重构：

### 1. 前端加载（内存与 IPC 瓶颈）

*   **当前交互**：选择文件 -> 读入前端内存缓存区 -> 转换为 Blob 发送。
*   **物理约束**：V8 Heap 单页内存上限、Tauri IPC 二进制传输灾难性序列化消耗。
*   **重构方案：Path-based Handoff (基于路径的轻量移交)**
    *   **拦截机制**：设定极低的前端内存警戒线（如 10MB）。超过极值的文件，前端**绝对禁止**调用 `readFile` 载入内存，仅通过 `@tauri-apps/plugin-fs` 的 `stat` 读取元数据（大小、类型、绝对路径）。
    *   **本地 Sidecar 优化**：由于 AI 引擎（Sidecar）运行于同机，前端向其投交导入请求时，彻底废弃 `FormData (Blob)`，改为下发**本地物理绝对路径（绝对路径透传）**。让底层的 Python 直接通过系统级 I/O 读取大文件，彻底绕过前端内存陷阱。

### 2. AI 解析（算力与时间瓶颈）

*   **当前交互**：发起 AI 解析请求（如 Whisper 音频提取） -> 挂起一个 loading 等待网络请求返回。
*   **物理约束**：GPU/CPU 算力与推理耗时，大型媒体动辄执行极长时间。
*   **重构方案：Asynchronous Job Queue (异步分离任务队列)**
    *   **判定分流**：根据文件类型或体积，判断属于“轻算力”（文本解析）还是“重算力”（大型视频）。
    *   **异步下发**：重算力任务投递后，Sidecar 立即返回 `202 Accepted`，带有任务 ID。前端解绑等待。
    *   **UI 移交**：在侧边栏引入“后台任务调度列表（Background Worker）”。应用主界面恢复可用。大文件在后台消化，完成后通过内部事件（类似于现在的 `slash:note-saved`）在右下角弹出成功通知，一键打开生成好的笔记。

### 3. 网络传输（带宽、网络脆性与大 Payload 瓶颈）

*   **当前交互**：提取变更清单 -> 将大二进制文件打入 JSON -> 将上百 MB 的 JSON POST 给同步服务器。
*   **物理约束**：HTTP 请求单次限制、网络抖动中断、Server 端解析几十兆 JSON 的内存灾难。
*   **重构方案：Two-Lane Sync Architecture (主副分离的双轨同步制)**
    *   **快车道（主同步链）**：保留原本的基于 JSON 差异的 Push/Pull。但**仅负责 Markdown 纯文本、元数据、以及体积 < 1MB 的超小图元**，保证文字灵感实现“微秒级”高频同步。
    *   **慢车道（重资产管道）**：剥离由于引入媒体而带来的大尺寸同步压力。引入二进制直传协议。
    *   **交互感知**：这部分必须脱离无感知的后台同步，变为显式的“传输管理器”。遇到大文件同步，在侧边栏顶部展现类似云盘的「传输列表」，不仅可以让用户直观看到大文件的具体进度，更从架构层面支持**断点续传（块传输）**和**暂停调度**。

### 4. 后端存储与云端边界（生命周期与管控边界）

*   **当前交互**：客户端代码硬编码判断 `TEAM_ASSET_MAX_SIZE = 50MB`。
*   **物理约束**：硬盘是用户的，带宽是用户的，但不归用户配置。
*   **重构方案：Server-Configured Governance (服务端下发治理)**
    *   **解耦硬编码**：移除任何客户端关于文件大小的绝对判断逻辑。
    *   **参数下发**：当用户登入服务器（或连接至服务端）时，服务器下发一份 `Capabilities` 清单。该清单由管理员通过 `.env` 或控制台配置，包含 `MAX_SYNC_SIZE_LIMIT`、`ALLOWED_MEDIA_TYPES` 等。客户端的“慢车道”严格遵守该规则预检。
    *   **UI 防范**：当大文件试图进入同步队列且超限时，阻断同步并给其在 UI 层面挂一个醒目的警告标（不再是悄悄丢弃），提示“本地资源超过服务器规定上限”。

## 实施阶段演进规划 (Phases)

**Phase 1: 防御性拦截（短期缓冲）**
*   维持 50MB 的限制逻辑作为临时防火墙。
*   将 `ImportHubModal` 的请求投递改为 Path 透传，化解立刻发生的前端 OOM 崩溃。

**Phase 2: 任务剥离与前端释放（中期攻坚）**
*   搭建“后台任务管线”，让所有超过 50MB / 音视频媒体的解析流程异步化。
*   不再卡死主操作流。

**Phase 3: 双轨制同步重构（长期战役）**
*   彻底拆解同步核心协议。
*   建立二进制断点直传机制与传输管理视图面板。
*   配置表下发，将大体量生态归还至管理员全权主导。
