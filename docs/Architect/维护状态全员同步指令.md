# 架构师指令 — 维护状态全员同步（最小方案）

> **签发**: 2026-04-23
> **适用范围**: OPT-03 维护模式全员感知
> **原则**: 零 DB Migration，搭车 negotiate 通道，服务端内存态

---

## 设计思路

维护模式是短时操作（通常几分钟），**不需要持久化到数据库**。服务端重启 = 自动退出维护，这是合理的。

数据流：

```
Admin 点击"开始维护"
  → 前端调用 POST /api/team/maintenance { vault_id, enabled: true }
  → 服务端在 AppState 的 HashMap<vault_id, bool> 中标记
  → 成员下次 negotiate 时，服务端顺带返回 is_maintenance: true
  → 成员前端读取后更新 Sidebar 标签
  → Admin 点击"退出维护" → 同理置 false
```

延迟 = 成员下次 negotiate 的间隔（AutoSync 30s 心跳 ~ 5min 保底），**可接受**。

```rust
/// 切换维护模式（Admin only）
pub async fn toggle_maintenance(
    State(state): State<AppState>,
    claims: Claims,
    Json(body): Json<serde_json::Value>,
) -> Result<Json<serde_json::Value>, ApiError> {
    let vault_id = body["vault_id"].as_str()
        .ok_or(ApiError::bad_request("missing vault_id"))?;
    let enabled = body["enabled"].as_bool().unwrap_or(false);
    
    // 权限校验：必须是 Admin
    crate::permission::ensure_team_permission(
        &state.pool, &claims.sub, vault_id, "manage"
    ).await?;
    
    // 写入内存态
    {
        let mut map = state.maintenance_vaults.write().unwrap();
        if enabled {
            map.insert(vault_id.to_string(), true);
        } else {
            map.remove(vault_id);
        }
    }
    
    tracing::info!(vault_id, enabled, "🔧 Maintenance mode toggled");
    Ok(Json(serde_json::json!({ "ok": true, "is_maintenance": enabled })))
}
```

### 第 4 层：前端

#### [MODIFY] `apps/desktop/src/services/AutoSyncManager.ts`

在 `sync:pulled` 事件 dispatch 附近（约 L661），读取 `result.is_maintenance` 并更新 store：

```typescript
// 在 dispatch sync:pulled 之后（约 L661）追加：
if (typeof result.is_maintenance === 'boolean') {
    const { useSessionStore } = await import('@/stores/useSessionStore');
    const currentState = useSessionStore.getState();
    // 仅非 Admin 时接受服务端的维护状态（Admin 自己控制本地状态）

        useSessionStore.setState({ isAdminManageMode: result.is_maintenance });
    }
}
```

```typescript
// 在 setAdminManageMode(!isAdminManageMode) 之后追加：
try {
    const config = syncService.getConfig();
    const teamVaultId = useSessionStore.getState().teamVaultId;
    if (config && teamVaultId) {
        const { teamService } = await import('@/services/TeamService');
        await teamService.request(config.serverUrl, config.accessToken,
            '/api/team/maintenance', 'POST',
```

## 行为预期

| 场景 | 结果 |
|------|------|
| Admin 点击"开始维护" | 本地立即显示 MAINTENANCE 标签 + 服务端标记内存态 |
| 成员下次 negotiate（30s~5min） | 收到 `is_maintenance: true`，Sidebar 显示标签 |
| Admin 点击"退出维护" | 本地立即清除 + 服务端清除内存态 |
| 成员下次 negotiate | 收到 `is_maintenance: false`，标签消失 |
| 服务端重启 | HashMap 清空，自动退出维护（合理行为） |
| Admin 调 API 失败（网络问题） | 不影响本地维护模式，下次重试。最差情况成员看不到标签，但数据安全不受影响 |

## 验证方案

1. `cargo check` 编译通过（server + desktop）
2. `vite build` 前端编译通过
3. Admin 开启维护 → 服务端日志打印 `🔧 Maintenance mode toggled vault_id=xxx enabled=true`
4. 成员客户端 30s 内 Sidebar 出现 `MAINTENANCE` 标签
5. Admin 关闭维护 → 成员客户端标签消失
6. 重启服务端 → 维护状态自动清除
