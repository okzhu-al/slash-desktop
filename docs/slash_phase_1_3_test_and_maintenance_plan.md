## 2. 架构回顾与代码清洁度指标 (Architecture Hygiene Review)

**在测试上述功能并进行 Bug 修复时，开发者必须秉持“童子军原则”，顺手对相关模块执行以下五大维度的审阅与重构：**

1. **架构合理性与逻辑解耦**
   - 检查长文件/“巨石块”（如过去的 `sync.rs`）是否按单一职责原则分叉。
   - 验证 Service 通信层与 React 视图渲染层是否完全分离。
2. **功能与页面的整合度（组件复用）**
   - 多场景存在的逻辑（如“团队用户列表”与“邀请管理”）应高度组件化抽象，消除代码的 Ctrl+C/V 坏味道。
3. **UI/UX 规范对齐**
   - 严格审查全局样式风格。遇到旧样式/不协调 UI，**务必引入并遵循项目根目录 `.agent` 的 `UI-UX-Pro-Max` 视觉引擎/设计规则**进行改造，保证应用保持统一的高级感与微交互体验。
4. **多端支撑与防线隔离**
   - 涉及网络、存储（`sqlite/fs`）的操作，需确保 Tauri Tauri CMD API 层具备完美的 Result Error 处理机制，为 Phase 4 进军 Tauri Mobile 铺平道路。
5. **代码清洁度与 i18n 完整度**
   - 零 Warning：Rust/TS 编译警告必须消除。
   - 国际化清算：任何硬编码的中文/英文文案（包括错误 Toast）在碰到之后立刻抽取加入 i18n 多语言命名空间中。

## 3. 维护文档生成规范 (Maintenance Documentation Protocol)

本次测试周期内，**凡是进行过核心 Bug 修复或重构的模块**，必须强制同步编写/更新维护文档。这是系统走向成熟规范化（Beta Release）的必经之路。

**规范格式参考：**

- **输出路径**：项目根目录下 `docs/maintenance/` 文件夹（需新建）。
- **文件命名**：`模块名_maintenance_guide.md`（例：`sync_v3_maintenance_guide.md`）。
- **标准章节结构**：
  1. **核心逻辑流水线 (Pipeline)**：该模块的工作机制是什么？(可用文本或 Mermaid 表达)。
  2. **状态机与核心数据 (State Definition)**：相关 DB 表、前端 Redux/Context 管理的字段及其意义。
  3. **已知雷区与防范 (Hazards & Edge Cases)**：本次 Bug 修复踩出了什么坑？如何避免复发？（如：Tldraw 的 SVG 裁剪、Token 过期时序等）。
  4. **修缮历史追踪 (Fix Log)**：记录关键修改的时间节点与 PR/Commits 大意。
