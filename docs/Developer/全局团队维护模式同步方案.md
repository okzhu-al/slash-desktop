# 全局团队维护模式同步方案

当前，Admin 的维护倒计时由于存储在 `useSessionStore`，是单纯的本地前端状态，其他团队成员无法感知。
为了实现"在所有成员的团队空间树 titlebar 上都显示出来"，我们必须跨越客户端边界，通过服务端同步这一状态。

## 现有系统约束
1. **无实时推送（无 Websocket/SSE）**：客户端目前依赖事件驱动（如本地文件落盘 500ms 后触发）或定时轮询（如协作红点每 30s、保底同步每 5m）来获取服务端信息。
2. **零配置同步局限**：Slash Team 仅同步用户加入的子目录（权限配置），不自动同步 `.slash` 根目录配置，因此不能简单地通过在团队目录下写入配置 JSON 来推送（会产生噪音）。

---

## 推荐架构设计

由于需要将这一状态以低延迟、高可靠性下发给全员并进行后续倒计时管理，推荐以下方案路线：

### 核心设计：数据库状态驻留 + 协商协议隐式下发

这是一种不需要客户端增加新的高频 HTTP 长轮询的方案，直接搭接在 Slash 已有的 Sync 服务骨架上，延迟取决于用户的编辑频率或后台心跳（~30s - 5m）。

#### [NEW] 后端存储扩展
- 创建 Migration: `017_vault_maintenance.sql`
- 在 `vaults` 表新增字段: `maintenance_expiry_at BIGINT DEFAULT 0`。

#### [MODIFY] 后端 API
- **触发入口**：在 `apps/server/src/routes/team.rs` 增加 `POST /api/team/maintenance` 接口。
  - Payload 接收 `{ vault_id, requested_duration_minutes }`。
  - 仅 Admin 能够调用成功。
- **协议注入**：修改 `slash-sync-proto` 中的 `SyncNegotiateResponse` 结构，增加 `Option<i64> maintenance_expiry_at`。
- **发送下发**：在 `apps/server/src/routes/sync/negotiate.rs` 中的每次协商请求处理时，提取 `vaults` 表中的 `maintenance_expiry_at` 一并放入 Response，借着高频的协商通讯“顺带”发送给所有前端。

#### [MODIFY] 前台联动响应
- **接收管道**：由于 `SyncService.ts` 原本就能接住协商响应，解析出这个新字段后，通过新的 `CustomEvent('sync:team-maintenance')` 广播。
- **倒计时修正**：在 `Sidebar.tsx` 中订阅该广播。一旦捕获到有效的 Timestamp，即刷新 `useSessionStore` 中的 `adminManageModeExpiry` 状态，触发秒级倒计时跳动。如果在未来，只要没过期，就能展示；过期自动消解。

---

## User Review Required

> [!WARNING]
> 因为当前系统没有 Websocket 长链接，客户端存在一定的滞后性（取决于触发最近一次协商或者 AutoSync 的时机，最差情况下会有数分钟落后）。
> **如果您需要在点击开启的一瞬间，其他成员的界面立即发生变化**，这就需要引入 SSE 或者是 Websocket 的基础建设（成本非常高昂）。
> 
> **请确认**：
> 1. 您是否同意进行 DB Schema 的更新（添加 `maintenance_expiry_at`）来支持真正的全域响应？
> 2. 这种通过已有协商通道「搭车」分发状态带来的几十秒的延迟是否不可接受，是否需要单独加一个前端通过 `setInterval` 高频拉取状态的专属 API？
