# Global Maintenance Mode Architecture Upgrade (OPT-03+)

您提出的“停止团队文件的其他来源变动”非常精准！这触及到了并发文件树维护的核心痛点。如果在管理员大范围调整目录结构时，其他成员依然能静默推送新文件、重命名或删除文件，将会导致严重的“幽灵目录”、“孤岛文件”以及“结构冲突还原”错误。

为了实现真正的跨网络全员冻结，我们需要将这套纯前端的“本地UI防线”升级为“**铁桶一样的后端系统锁**”。

## User Review Required

> [!WARNING] 
> 增加全局锁需要修改数据库结构，并可能对正在进行中的数据同步发生硬性阻断。请审核以下的方案。

## Proposed Changes

### 1. 数据库升级层 (Backend Database)
新建一个 SQL migration 文件，用于为 postgres 记录当前的全局维护状态。

#### [NEW] `apps/server/migrations/017_vault_maintenance.sql`
```sql
ALTER TABLE vaults ADD COLUMN is_maintenance BOOLEAN NOT NULL DEFAULT FALSE;
```

#### [MODIFY] `apps/server/src/models.rs`
在 `VaultRow` 结构体中添加对应的字段映射。

### 2. 权限与状态控制器 (Backend APIs)
我们需要给后端拓展切换管理状态的功能。目前的前端是本地自嗨切换，我们需要通过网络请求告诉服务器。

#### [MODIFY] `apps/server/src/routes/team/core.rs` (或新建专门入口)
- **增加端点** `POST /api/team/{vault_id}/maintenance`。
- **校验逻辑**：严格检查 JWT Token 以及当前操作者在 `team_members` 中是否拥有 `Admin` 权限。
- **持久化**：将对应空间的 `is_maintenance` 字段进行布尔值切换。

### 3. 全局拦截器网关 (Backend Interceptors)
修改涉及空间改动的关键后端写入侧逻辑。核心思想：**当空间进入维护期时，仅放行处于维护状态中的管理员执行管理员特性。**

#### [MODIFY] `apps/server/src/routes/sync/negotiate.rs` / `push.rs`
- **Promote / Push 阻断**：在同步协程的验证阶段，检查当前 Vault 的 `is_maintenance` 字段。若开启维护通道，抛出 `HTTP 423 Locked`，直接终止同步管线（即拦截前端发来的 `Promote` 和后续更新）。

#### [MODIFY] `apps/server/src/routes/team/directories.rs` 及 `files.rs`
- **Rename / Delete 拦截**：增加对普通写入和删除操作的拦截判断，保障只有 Admin 在清理期可正常使用。

### 4. 客户端双向互通层 (Frontend Sync & UI)

#### [MODIFY] `apps/desktop/src/features/team/TeamManagePage.tsx`
- 点击“启动系统维护”按钮时，将发起真实的网络请求（调用 Axios 到后端的新端点）。
- 成功后，才做 `setAdminManageMode(!isAdminManageMode)`。

#### [MODIFY] `apps/desktop/src/services/TeamService.ts`
- 增加通信函数 `toggleMaintenanceMode(vaultId: string, status: boolean): Promise<void>`。

## Open Questions

> [!CAUTION] 
> **锁定粒度问题**：当一名 Admin 开启维护模式时，其他也拥有 Admin 权限的联合管理员是否也应当被禁掉 Promote 等普通写入操作？（推荐方案：维护期间关闭对该空间的各种写入端点。仅有“超级特权的 /admin_rename”、“/admin_delete”能通过。您同意此一刀切防并发的强手段么？）
