# OPT-05/06 阶段四 + 阶段五：Zero-Shot 本地转写与多媒体语义织网 — 研发落地报告

> **编制**: 核心开发者
> **日期**: 2026-04-28

## 一、项目概述

本次研发成功落地了 Slash 系统的多媒体 AI 分析管线，涵盖两个核心模块的大规模重构与增强：
1. **Sidecar 音视频转写引擎升级 (Phase 4)**：使用离线、高精度的 `faster-whisper` 彻底替换了原先依赖互联网且不稳定的 Google Web Speech API。
2. **Rust 多媒体语义织网 (Phase 5)**：通过在调用 Embedding 模型前的预处理拦截，将笔记中的多媒体资产（如音视频和图片的内嵌引用）进行本地直读解析，使得 AI 可以真正“看见”多媒体内容，极大增强了语义搜索与 GhostLink 的召回精度。

---

## 二、架构改动与实现细节

### 2.1 零样本离线转写引擎 (`apps/python-sidecar`)
* **完全剥离外部依赖**：删除了 `speech_recognition` 及关联的动态 Monkey Patch (`audio_lang_ctx`)，消除了网络不稳定引发的超时阻塞和转录失败问题。
* **懒加载与资源优化**：在 `main.py` 中引入了 `get_whisper_model()` 单例模式，仅在系统首次处理音视频流时加载 `faster-whisper` 的 `tiny` 模型（采用 `int8` 精度和 CPU 加速运算）。
* **格式分流与直写协议**：在 `/parse` 端点的前置链路上检测格式后缀（`AUDIO_VIDEO_EXTENSIONS`）。若是音视频格式，则利用临时文件 (`tempfile`) 分发给 Whisper 引擎，引擎原生侦测源语言，完成后直接构建 `ParseResponse` 提前返回，大幅降低了不必要的文本管道开销。

### 2.2 多媒体语义融合管线 (`apps/desktop`)
* **正则预检与路径映射**：在 `src-tauri/src/commands/ai/skills.rs` 中，为 `trigger_ai_skill` 添加了前置处理步骤，利用正则表达式 `!\[.*?\]\((assets/[^)]+)\)` 探测笔记中的多媒体资产。
* **本地微服务协同**：使用 `reqwest` 向 Sidecar `http://localhost:3722/parse` 接口发起短轮询（120s 超时保障），并利用阶段二/三完成的**路径直读模式** (`local_path`) 读取磁盘资产，无缝提取视频旁白、音频转写或图片 OCR 文本。
* **高维向量织网**：将提取到的富文本内容进行硬截断（单文件最多截取 2000 个有效字符），按 `[Media: xxx]` 格式统一附加在笔记正文尾端 (`enriched_content`)，共同投喂给 Ollama 的 Embedding 管道，实现了文本与多媒体在同一维度的向量化空间降维。

## 三、关键防御与降级设计

1. **静默降级策略**：若 `reqwest` 获取多媒体文本超时或 Sidecar 容器宕机，系统仅通过 `log::warn` 抛出警告，保留原文本并继续生成纯文本的 Embedding，**不阻断 AI 管线运行**。
2. **内存风暴遏制**：严格限制多媒体抽取上限（2000 Char/文件），防止恶意超长视频转写撑爆本地 Ollama 的显存/内存 `Token Window`。
3. **线程捕获隔离**：在 `skills.rs` 内部，通过作用域隔离和 `move` 闭包安全地转移了 `vault_path_str` 生命周期，防范了多线程并发环境下的数据竞争 (Data Race) 以及借用冲突。

## 五、部署与上线须知

因本次 Sidecar 引擎彻底更迭了核心的音视频底层依赖库，在更新前后需执行以下指令重建 Python 运行环境：
```bash
docker compose build sidecar && docker compose up -d sidecar
```
*注：由于首次转写会自动拉取 CTranslate2 和 `tiny` 模型权重文件（约 100MB），首次执行解析时可能会有几秒钟的网络波动延迟，后续将完全转为毫秒级本地预测。*
